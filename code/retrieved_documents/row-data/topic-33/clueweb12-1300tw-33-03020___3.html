As promised here is my answer, please note these are my views only and I advise you do your own testing and research.

Having worked with all the technologies mentioned (bar JSP); here is the breakdown:

I would like to make two groups:

Group 1 (Open source; LAMP stack):

This group is the PHP and MySQL combination, and will be typically run under Linux based servers. Of course this combination is not strictly the only combination. One example could be a WAMP stack: using a Windows based OS with PHP and MySQL.

Of course PHP can be used with any SQL solution, and could be used with Microsoft SQL Server. However this is not the norm.

In this group there is also the "Java stack" JSP and it is also cross platform.

Group 2 (Closed source; MS stack):

In this group are Classical ASP and ASP.NET along with MS SQL server. Once again, ASP.NET can also be used with MySQL (as MySQL is cross platform). However this is a rare combination.

Although via the "Mono" project one can technically run ASP.NET platform under Linux, its not a very popular combination.

Your first choice should depend in what you want to work with, open source (free) or closed source (paid).

Once you have made this choice, here are the details of each technology from my perspective:

PHP and MySQL

It is cross platform and free. In terms of PHP from a development perspective it has a "Double Edged" sword effect:

In one sense it is incredibly easy to do common things (90% of most web application functions, eg send email, connect to database, parse data etc).

However this "ease" of use comes at a major cost, the Language is a total failure. If you look at the history behind the birth of PHP one can understand why this is the case. In fairness, PHP has come a long way and has strived in each release to "improve" the many many language design issues.

However it is an uphill struggle, because it started of from the wrong foot!

The other problem is one of "barrier to entry"; because its so "easy" to "learn" PHP. This has meant that a lot of "non developers" or people who have NO formal training as professionally programmers get into "web development".

The results have been a disastrous, the industry has been effected in a NEGATIVE way, of course one can not put 100% blame on PHP alone it is far more complex than that.

It is for this reason that most professional developers look with "disapproval" when it comes to PHP.

The other issue is that security is not "default", in other words one has to actively know what they are doing to avoid serious security issues.

The other problem is "Spaghetti Code" this is when SQL, Business Logic, and Presentation Logic are all mashed into one place. This is a huge problem because this reduces maintainability; readability and makes it very difficult to extent etc. It takes GREAT care and experienced and highly competent developers to create very clean and highly maintainable code.

Perhaps it is for this very reason that as of late many "Web Frameworks" have become so incredibly popular. They are created to solve this very issue of "separation of concerns".

However frameworks, invariably become yet another layer of abstraction. Adding a new layer of abstraction will mean that as long as what you have in mind is within the strictures of the framework then all is possible. However if you require to do something that is outside of the scope of the framework; then you will have to battle with the framework in order to do what you want to achieve.

In Summary:

Quote: Using PHP, is like driving a fast racing car, without any safety gear, no helmets, no harness. It takes a master driver to race such a car.

Inexperienced drivers will go fast, however they will also likely crash it and die in a blazing inferno.

ASP.NET and SQL Server

ASP.NET (which should not be confused with classic ASP) are both Microsoft propriety closed source web technologies.

Classic ASP is outdated and could be thought of as Microsoft’s first attempt at web server side technology. It is script based(Visual Basic or Jscript) and executes on the server.

I will not discuss further regarding classic ASP since it is really already legacy tech, and unless you have a very specific reason to learn it (supporting legacy system) then you are advised against it.

In order to understand ASP.NET and how its related eco system fit in, you first need to understand what “.NET” is.

“.NET” in a one-liner explanation can be thought of as Microsoft’s “answer” to Java. If you are aware of how conceptually Java works then “.NET” has many similar parallels.

.NET is a very large topic and I can’t do it justice in this small write-up and I highly suggest you read up on this topic for further information. However I can give you a brief idea.

.NET has two parts, first there is the Virtual Machine called the CLR(Common Language Runtime). The second part is the BCL(Base Class Library). The BCL provides a common set of libraries to handle the usual software tasks such as graphics processing and file handling.

The .NET platform allows multiple languages to target the CLR, this is done by first compiling source could to MSIL(Microsoft Intermediate Language), which can then be turned into highly optimised CLR byte code.

In theory at least, not only is the .NET platform language agnostic(having over 40 languages) but also OS independent (via mono project). However in reality it is almost always run in a Windows environment.

ASP.NET is simply a “Web Framework” that uses the .NET platform to power it. This means one can share code across different domains(example, desktop application classes could be used in web application projects and vice versa).

Unlike PHP and other server side scripting technologies, ASP.NET are source compiled, which means that they are very efficient for creating CPU demanding web applications.

ASP.NET currently comes in two “camps”, one is the “established winforms” and other is the fairly newer “MVC” ala Ruby/Python inspired web framework.

Both use the exact same underlying technology, however are abstracted differently. In the WinForms world, the abstraction is an event based paradigm(those coming from a desktop programming environment will be very familiar with this). As such WinForms developers must understand “Page Cycles” in order to effectively develop web application.

ASP.NET provides a whole host of advanced features such as Custom Controls, Global .asax handlers, Generic Handles, Master Pages, and many other features to build enterprise level systems. In terms of security generally speaking unless the developer is extremely “sloppy” than it is fairly secure “out of the box”.
In terms of tooling, it is really mature and feature complete. Almost the entire development can be done from inside of Visual Studio(the .NET IDE).

However it is not all “sunshine and rainbows”. There are also many downsides: It is not really cross platform, picking ASP.NET means you are effectively tied to the Microsoft Platform. It is paid software; you will need to pay for licences for commercial uses.

It is not open source, and Microsoft has an aggressive development cycle, constantly updating and changing the technologies.

It’s far more complex, sometimes it may even be overkill if you want to do simple things. In such cases, PHP or even classic ASP would be much better suited.

In summary:

Quote: ASP.NET is like an advanced spacecraft, it has all the advanced bells and whistle.
But if you want to do something really fast and really simply and for free then this is not it.
